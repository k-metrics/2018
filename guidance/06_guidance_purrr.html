はじめに

purrrパッケージ

  • purrr = purr + R らしい
    • 関数型プログラミングを実現するためのパッケージ
  • tidyverseファミリーでtidyなアーキテクチャに基づいた設計
  • apply関数群やplyrパッケージを再設計したようなパッケージ
    • 層別的な処理や反復処理に威力を発揮する
  • 指定するデータ（.xや.y）に対して指定する処理（.f）を適用する
    • 指定できるデータはベクトル型、または、リスト型
    • 指定できる処理は関数に限らず演算子でも可能
  • はっきり言って慣れないと難しい
    • 無名関数やラムダ式といった聞きなれない言葉が出てくる
  • 使うとすごく便利
    • 特にリスト型変数の処理はめちゃくちゃ楽になる

リスト型データの展開

対象データ（Redmineチケット）

既に学んだようにredmineRパッケージで取得したデータはリスト型変数ばかりで処理が面倒。

issues %>% head()

　
それでもCSVファイルに出力して処理するよりは楽。

リスト型データの中身

リスト型データからはidかnameが取得できれば可視化・分析につかえる。

issues %>% head(5) %>% dplyr::select(id, project, tracker) %>% knitr::kable()
id project tracker
29767 list(id = 1, name = “Redmine”) list(id = 3, name = “Patch”)
29764 list(id = 1, name = “Redmine”) list(id = 2, name = “Feature”)
29763 list(id = 1, name = “Redmine”) list(id = 2, name = “Feature”)
29758 list(id = 1, name = “Redmine”) list(id = 2, name = “Feature”)
29757 list(id = 1, name = “Redmine”) list(id = 2, name = “Feature”)

そこで、purrr::map

issues %>% head() %>% dplyr::select(id, tracker) %>% 
  dplyr::mutate(tracker_name = purrr::map(tracker, 'name'))

　
list[2]がchr[1]になってる！なんか期待できるかも？

なにをやってるの？

purrr::map関数は以下のような引数を取る関数で返り値はリスト型です。

purrr::map(.x, .f, ...)

第一引数（.x）に対して第二引数（.f）で指定した処理（関数や演算子など）を適用する関数です。前ページの処理では、.fに名前（“name”というリスト内の要素名）を指定していますのでインデックス参照が行われています。
　
すなわち、

issues$tracker[[n]]$name

という参照を行っているのと等価の処理が行われています（n = 1, 2, …）。
　
参考までにベクトルやリストなどの参照に使われる[や[[、$は演算子の一種（要素アクセス演算子とでもいうか）です。

でもリストのままじゃ嫌だ

purrr::map関数は前述のように返り値がリストです。ベクトルの返り値が欲しい場合にはpurrr::map_関数を用います。“name”は文字列なので、ここではpurrr::map_chr関数を用います。

issues %>% head(5) %>% dplyr::select(id, tracker) %>% 
  dplyr::mutate(tracker_name = purrr::map_chr(tracker, 'name'))

例外処理

ただ、チケットデータには以下のように欠損値（NA）が含まれています。

issues %>% dplyr::select(id, assigned_to, category) %>% head(4) %>% knitr::kable()
id assigned_to category
29767 list(id = 332, name = “Go MAEDA”) list(id = 12, name = “Translations”)
29764 NA list(id = 2, name = “Issues”)
29763 NA list(id = 8, name = “Administration”)
29758 NA NA

　
（つづく）

例外処理（つづき）

これをpurrr::map_関数群で処理するとSTOPエラーとなり処理が完了できません。

issues %>% dplyr::select(id, assigned_to, category) %>% head(4) %>% 
  dplyr::mutate(assignee = purrr::map_chr(assigned_to, 'name'),
                cat_name = purrr::map_chr(category, 'name')) %>% knitr::kable()
## Error in mutate_impl(.data, dots): Evaluation error: Result 2 is not a length 1 atomic vector.

　
欠損値を除外して処理するためにはpurrr::map_if関数を用います。purrr::map_if関数は第二引数（.p）で条件式を指定します。

purrr::map_if(.x, .p, .f, ...)

　
（つづく）

例外処理（つづき）

issues %>% dplyr::select(id, assigned_to, category) %>% head(4) %>% 
  dplyr::mutate(assignee = purrr::map_if(assigned_to, !is.na(assigned_to), 'name'),
                cat_name = purrr::map_if(category, !is.na(category), 'name'))

　
purrr::map_if関数も返り値はリストなのベクトルの返り値が欲しい場合は、もう一段、処理をする必要があります。どうしたらよいでしょうか？
　
（つづく）

例外処理（つづき）

issues %>% dplyr::select(id, assigned_to, category) %>% head(4) %>% 
  dplyr::mutate(assignee = purrr::map_if(assigned_to, !is.na(assigned_to), 'name') %>%
                  purrr::map_chr(1L),
                cat_name = purrr::map_if(category, !is.na(category), 'name') %>%
                  purrr::map_chr(1L))

　
このような処理は、変換したいベクトルの型に合わせて関数化しておくと便利です。

展開結果の例

今までの処理を適用すると下記のようなチケットデータを得ることができます。
　

カスタムフィールドの展開

カスタムフィールドとは？

Redmineではカスタムフィールドと呼ばれるユーザが自由に定義できる項目（フィールド）があります。カスタムフィールドはプロジェクト毎、トラッカー毎に異なる設定ができるため、redmineRパッケージで取得できる情報ではcustom_fieldsという項目にリスト型変数がネストされた形で格納されます。

カスタムフィールドをRのコードで表現すると下記のようなイメージになります。

custom_fields <- c(
  list(list(id = n, name = "character", value = c("character", ...)),
       list(id = n, name = "character", value = c(m, ...)),
       ...),
  list(list(id = n, name = "character", value = c("character", ...)),
       list(id = n, name = "character", value = c(m, ...)),
       ...), ...
)

リストの中身

ネストされているリスト型変数内の要素は以下のような意味を持っています。
　

要素 内容
id カスタムフィールドの識別子（設定時にシステム側で自動付与）
name カスタムフィールド名称（「名称」欄に設定した値）
value カスタムフィールドの値（単一とは限らない）

　
valueはカスタムフィールドの設定により取れる値が変わります。

実際のカスタムフィールド

ネスト構造を解消する

手始めに他のリスト型変数と同様の形にするためにtidyr::unnest関数を用いてネスト構造を解消します。この時に識別ができるよう識別番号（チケット番号）と共に処理してください。

issues %>% head(4) %>% dplyr::select(id, custom_fields) %>% 
  tidyr::unnest() %>% knitr::kable()
id custom_fields
29767 NULL
29764 list(id = 2, name = “Resolution”, value = "")
29763 list(id = 2, name = “Resolution”, value = “Invalid”)
29758 list(id = 2, name = “Resolution”, value = "")

欠損値を削除する

次の展開に際して欠損値（NA）は不要ですので、これらを削除しておきます。

issues %>% head(5) %>% dplyr::select(id, custom_fields) %>% 
  tidyr::unnest() %>% tidyr::drop_na() %>% knitr::kable()

issues %>% 
  dplyr::filter(id == 16451 | id == 16447) %>% 
  dplyr::select(id, custom_fields) %>% tidyr::unnest() %>% 
  with(., purrr::map_df(custom_fields, c))

　
これを二変数に拡張できればインデックスとなるチケット番号とカスタムフィールドの関係表が作れるはず。

二変数を扱うpurrr::map2関数群があるじゃないか！

issues %>% 
  dplyr::filter(id == 16451 | id == 16447) %>% 
  dplyr::select(id, custom_fields) %>% tidyr::unnest() %>% 
  with(., purrr::map2_df(.x = id, .y = custom_fields,
                         .f = function(.x, .y) {c(issue_id = .x, .y)}))

（つづく）

ワイド型に変換してカスタムフィールドを除いて展開したチケットデータと結合（dplyr::*_join）すれば完成。

「""」をNAに変換する

気が付いた方もいらっしゃるとは思いますが、value要素が「""」になっていた部分はNULL扱いになっていますのでdplyr::na_if関数を用いてNAに統一します。
